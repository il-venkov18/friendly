# Руководство по разработке проекта friendly

## 1. Git Workflow

Мы придерживаемся git-flow. Структура веток:

- `master` — стабильная продакшн-ветка
- `dev` — основная ветка разработки
- `feature/<название>` — новые фичи
- `hotfix/<название>` — срочные исправления

### Правила:
- В `master` и `dev` запрещены прямые коммиты
- Все изменения проходят через Merge Request
- Требуется ревью хотя бы от одного участника команды


## 2. Conventional Commits

Мы используем [Conventional Commits 1.0.0](https://www.conventionalcommits.org/en/v1.0.0/).

**Формат коммита:**
<type>(optional scope): <description>

**Примеры:**
- `feat(auth): добавлен экран входа`
- `fix(profile): исправлена ошибка при загрузке аватара`
- `chore(deps): обновлены зависимости`

**Допустимые типы:**
- `feat` — новая функциональность
- `fix` — исправление бага
- `docs` — изменения только в документации
- `style` — изменения в форматировании (пробелы, отступы и т.п.), без изменения кода
- `refactor` — правка кода без добавления фич или исправлений багов
- `test` — добавление или изменение тестов
- `chore` — изменения в конфигурации, инфраструктуре, зависимостях


## 3. Стиль кода

- **Компоненты**: `PascalCase`, например: `UserCard.tsx`
- **Утилиты и хуки**: `kebab-case`, например: `use-toggle.ts`, `format-date.ts`
- **Типы**: `feature-name.types.ts`, например: `auth.types.ts`
- Один компонент = один файл
- Неиспользуемые экспортируемые типы и функции не оставляем в коде
- Используем `ESLint` и `Prettier` (подключаются по мере стабилизации проекта)


## 4. Переменные окружения

Перед запуском создайте файл `.env` на основе соответствующего `.env.example`.

- Для frontend: `frontend/.env.example`
- Для backend: `backend/.env.example`

Файл `.env` не должен попадать в репозиторий — убедитесь, что он указан в `.gitignore`.
